# Alexandria-Hermes 개발 규칙

## 목적

Alexandria-Hermes를 안정적인 장기기억 Backend로 개발하고 리팩토링하기 위한 순수 개발 규칙이다.

이 문서는 PRD가 아니다. 특정 기능, 구현 순서, 프로젝트 우선순위를 자동으로 결정하지 않는다.

## 핵심 원칙

- 기존 구조를 먼저 조사하고 재사용한다.
- Obsidian Markdown을 Canonical Storage로 유지한다.
- SQLite, FTS, Vector, Embedding, Graph Index는 재구축 가능한 Read Model 또는 Index로 취급한다.
- 검증이 필요한 외부·저장 경계에는 Pydantic v2를 사용한다.
- 검증이 끝난 내부 DTO에는 Dataclass를 사용할 수 있다.
- 알려진 Dictionary Shape를 생 `dict`로 전달하지 않는다.
- 부분적·Raw Mapping에는 TypedDict를 사용한다.
- Production Code에서 Any와 불필요한 Optional을 금지한다.
- 기능과 직접 관련된 Touch-path만 리팩토링한다.
- 코드 존재와 실제 검증을 구분한다.
- Lint, Typecheck, Test를 통과하지 않으면 완료를 주장하지 않는다.

## 모델 역할 분리

```text
FastAPI / MCP / File / JSON / Frontmatter Input
→ Pydantic Validation Contract

검증 후 내부 계층 간 전달
→ frozen Dataclass Internal DTO

SQLAlchemy
→ Persistence Model

Raw 또는 부분 Mapping
→ TypedDict

동적 JSON 경계
→ 제한된 JsonValue 계열 Type

Obsidian Markdown
→ Canonical Storage

SQLite / FTS / Vector / Graph
→ Rebuildable Index
```

동일한 객체를 이유 없이 Pydantic, Dataclass, ORM Model 세 종류로 중복 정의하지 않는다. 각 표현이 서로 다른 책임을 가질 때만 Mapper를 둔다.

## 완료 상태

- `DISCOVERED`: 조사 완료
- `PLANNED`: 변경 범위와 검증 방법 확정
- `IMPLEMENTED`: 코드 작성 완료, 검증 미완료
- `VERIFIED`: 요구된 품질 명령과 테스트 통과
- `BLOCKED`: 환경·선행 결정·기존 결함으로 검증 불가

테스트를 실행하지 않았으면 `VERIFIED`를 사용할 수 없다.
